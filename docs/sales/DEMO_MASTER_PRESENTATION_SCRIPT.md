# SchoolVault Demo Master Presentation Script

Use this as the single presenter document for school demos.

---

## 1) Demo Objective (say this first)

"Today we will show how one school can run academics, operations, communication, and parent engagement from one secure platform — with role-based access, tenant isolation, and day-to-day workflows that reduce manual work."

---

## 2) Demo Run-Time Tracks

- **10 min track**: leadership preview (principal + management)
- **20 min track**: decision demo (principal + admin + operations head)
- **40 min track**: full product walkthrough (principal + section in-charge + bursar + operations)

If time is short, always keep this order:
1. Outcomes dashboard -> 2. Daily school operations -> 3. Parent experience -> 4. Controls and governance.

---

## 3) High-Impact Demo Storyline (recommended order)

## A. Opening impact (2 min)

Say:
"Most schools lose time in follow-ups, fragmented tools, and role confusion. We solve this with one workflow system where each role sees only what they need, and management gets live visibility."

Show:
- Dashboard
- Notification widgets
- Quick proof of school/tenant context

## B. Academic core (3-5 min)

Show:
- Academic year, class/section setup
- Students list
- Attendance and timetable

Say:
"Academic structure is the base layer. Every downstream module — exams, fees, parent portal — respects this same structure."

## C. Exams and publish-control (3-4 min)

Show:
- Exam cycle
- Exam timetable
- Marks entry/publish behavior

Say:
"Schools control exactly when marks become visible to parents. Draft and internal review stay private until publish."

## D. Fees and collection visibility (3-5 min)

Show:
- Fee structures
- Payment records/defaulter indicators
- Collection summary

Say:
"Finance team gets clear collection visibility, and leadership gets trend visibility for risk planning."

## E. Operations modules (7-10 min)

### Transport
- Route, exceptions, operational queue
- Explain optimization and exception handling

### Hostel
- Occupancy analytics
- Incident + SLA escalation policy
- Gate pass / visitor desk actions

### Library
- Catalog -> issue/return
- Reservation + fine policy + inventory adjustments

Say:
"Each operational module is workflow-first: clear desk actions, escalation rules, and management analytics."

## F. Parent/portal experience (3-5 min)

Show parent login:
- Child-scoped data
- Fees, attendance, exam visibility
- Teacher communication lane

Say:
"Parent trust improves when updates are timely and child-scoped — no noise, no wrong data exposure."

## G. Governance and control (2-4 min)

Show:
- Role capability and permission controls
- Admin delegation concept

Say:
"Admin can delegate by duty — teaching and non-teaching staff can be granted only the required read/write actions."

---

## 4) School-Size-wise Demo Variant

## Small school (<=1,000 students)

Focus:
- Simplicity, fewer clicks, single-desk operations
- Combined staff role packs

Key line:
"One admin team can run all major workflows from one panel."

## Mid-size school (1,000-3,000 students)

Focus:
- Multi-role delegation
- Exceptions and escalations
- Parent communication reliability

Key line:
"As the school grows, process discipline stays consistent without adding chaos."

## Large school/group (3,000+ students or multi-campus intent)

Focus:
- Tenant-safe architecture
- Role granularity
- Analytics and policy-driven controls

Key line:
"This is built for scale with strict data boundaries, role segregation, and operational observability."

---

## 5) Presenter Login Sequence (recommended)

1. School Admin (control + overview)
2. Teacher (execution reality)
3. Parent (trust + experience)
4. Optional: Library/Hostel/Transport desk persona (ops depth)

Do not start with parent view; start with admin impact.

---

## 6) What to verbalize while clicking (quick cues)

- **When opening any screen**: "This is role-scoped and school-scoped."
- **When doing actions**: "Actions are auditable and permission controlled."
- **When showing analytics**: "This is decision data, not just reports."
- **When switching role**: "Same platform, different responsibility views."

---

## 7) Likely school questions and best answers

## Q1: "Can we delegate modules to non-teaching staff?"
Answer:
"Yes. Admin assigns fine-grained permissions by responsibility, not by broad role only."

## Q2: "Will parents see data before approval?"
Answer:
"No. Publish-controlled flows ensure internal draft/review is hidden until approved."

## Q3: "Can this handle our growth?"
Answer:
"Yes — it is multi-tenant safe, role-based, and built around scalable workflow modules."

## Q4: "Can we run only selected modules first?"
Answer:
"Yes. You can phase rollout module by module and expand without platform migration."

## Q5: "How fast can staff learn this?"
Answer:
"UI flows are desk-oriented and role-specific. Most staff are productive after short guided onboarding."

---

## 8) Objection handling (business-focused)

- **"Too many features"** -> "We rollout by phase, not all at once."
- **"Data/privacy concern"** -> "Role + tenant + workflow controls enforce scoped visibility."
- **"Will adoption fail?"** -> "Daily workflows are mapped to existing school operations."
- **"We already use Excel/manual systems"** -> "This keeps your process intent but removes reconciliation delays."

---

## 9) Recommended close (final 2 minutes)

Say:
"If useful, we can run a pilot with your real workflow priorities — attendance + exams + fees first, then operations modules in phase 2."

Ask:
- "Which 3 workflows cause the most friction in your school today?"
- "Who should be the pilot owner on your side?"
- "Do you prefer a 2-week guided pilot or a staged month rollout?"

---

- Seed data visible in key modules
- Parent account linked to children
- Internet + backup hotspot
- One fallback screenshot deck
- Keep total demo <= promised time
